构造graph--->
      1. 加载的文本str转list
      2. 调用mdhash为text生成唯一"doc-id"
      3. 拿着“doc-id”去数据库比对去重
      4. 对不在数据库中的text切块
            (1). 对text进行encode编码
            (2). 按照token_size切块
      5. 对切块后的chunk调用mdhash生成"chunk-id"
      6. 拿着"chunk-id"去数据库中比对去重
      7. 对不在数据库中的chunk进调用LLM进行关系提取
            (1). 写提示词prompt
            (2). 将prompt+text文本内容合并成一个大的hint_prompt喂给LLM，得到result
            (3). 将hint_prompt和result打包成标准的openai对话格式
            (4). 根据提前设定的LLM提取次数，进行重复提取，榨取知识，得到final_result
            (5). 对LLM输出的final_result进行处理，分别得到四元节点信息（entity_name, entity_type, description, source_id）和五元边关系（src_id, tgt_id, weight, description, source_id）
            (6). 对上述得到的节点信息和边信息进一步处理，主要判断图中是否已经有该节点和边，如果没有插入，如果节点已经存在，则收集节点的(entity_type, description, source_id)然后将其与已有的合并；如果合并的description描述太长已经超过summary_max_tokens，则调用LLM生成新的摘要并更新；如果关系边已存在，则收集(weight, source_id, description, order)，其中order是使用DSpy进行实体提取时才会返回，在合并时候如果关系有order则取所有的最小值，如果没有，默认值为1.
      8. 在得到新的实体关系后，使用leiden算法进行聚类
      9. 生成最终的报告。
            (1). 获取所有社区的schema信息，按照level从高到低排序。level越大，社区越小，粒度越细
            (2). 先生成小社区报告，然后再生成大社区报告。并且大社区报告生成时可以引用已经生成好的小社区报告作为输入。
            (3). 单个社区报告生成核心：
                  (a). 从图存储中获取社区的所有节点和边数据
                  (b). 按节点或者边的度数进行排序，保留重要的节点/边，并控制token的预算长度
                  (c). 将上述打包好的描述加上提示词模板，调用LLM生成结构化json报告
                  (d). json转为markdown格式的可读文本
                  (e). 报告持久化存储到KV存储中

GraphRAG.aquery(query, param)
├── param.mode == "global"
│   └─→ global_query()                         [utils.py]
│         ├─→ knowledge_graph_inst.community_schema()  [base.py]
│         ├─→ community_reports.get_by_ids()           [base.py]
│         │
│         ├─→ _map_global_communities()         [utils.py]  ← Map阶段
│         │     ├─→ truncate_list_by_token_size()        [utils.py]
│         │     └─→ best_model_func()                    [llm.py]
│         │
│         └─→ best_model_func()                 [llm.py]  ← Reduce阶段


查询
      1. 获取社区并过滤，根据level只保留低于设定level的社区
      2. 根据occurrence（出现频次）重要性进行降序排序，只取前k个社区
      3. 用社区的ID列表去KV存储中获取报告数据，并按照rating过滤，rating是社区重要度，之后再次根据(occurrence, rating)双关键字进行降序排序，确保最重要的社区排在最前面
      4. MapReduce阶段
            (1). 将社区报告分组，每组独立调用LLM提取与用户问题相关的关键点
                  (a). 将社区报告按token预算切分成多个组，保证每个组的报告输入给LLM不会超过模型的上下文窗口
            (2). 并行调用LLM输出
                  (a). 将社区报告格式化为csv表格，有4列：id, content, rating, importance
                  (b). 调用LLM输出json格式的points，包括description和score
            (3). 返回结果是一个list， 即[[point1, point2], [..] ]
            (4). Reduce阶段：
                  (a). 过滤：丢弃没有descrptiond字段的point， 丢爹score<=0的point
                  (b). 排序：按照score降序排序，最重要的point排在前面
                  (c). 截断：控制输出的总token，防止LLM输出过长
